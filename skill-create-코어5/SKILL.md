# /skill-create — 스킬 제조 공장

> **PO의 철학:**
> "가져다 쓰지 마라. 분해하고, 결함 부분은 걸러내고, 검증된 부품만 가려내 조립하라.
>  발굴 자체가 실력이 아니라 — 발굴 + 검증 + 자기 자산화까지 가는 흐름이 실력이다."

**필요한 Claude Code 스킬을 막연한 요청 한 줄에서 받아, 완성·검증된 `SKILL.md`로 출하하는 메타 스킬 — 스킬을 만드는 스킬.**

> **⚠️ 제조 대상 한정 (V3.2):** 이 스킬은 **Claude Code 스킬** — `~/.claude/skills/{name}/SKILL.md` 형태로 등재되어 슬래시 명령(`/{name}`)으로 호출되는 절차·방법론 단위 — 만 제조한다. LLM 의존형 독립 에이전트(웹사이트·데몬·API 등 자체 실행체) 제조는 형제 공장 `/llm-dependent-agent-create`가 담당한다 (둘은 같은 9-Phase 방식·같은 "분해·검증 후 조립" 철학을 공유하며, 같은 자산이 스킬로 출하될 수도 에이전트로 재포장될 수도 있는 양방향 관계다).

이 스킬은 공개 저장소의 기존 스킬을 **통째로 신뢰하지 않는다.** 발굴물을 작업대에 올려 부품(Phase 블록·프롬프트·체크리스트) 단위로 검증하고, 안전하지 않은 부분을 걸러낸 뒤 검증된 부품만 적용한다. 그 위에 자체 작성을 얹어 조립하고, QC 출하검사를 통과한 것만 자산으로 등재한다. **출하 후에는 운영 루프**가 돌면서 사용 통계·피드백을 Phase 1로 회귀시켜 다음 사이클을 시작한다.

## 공장 비유 (Manufacturing Metaphor)

```
원자재 입고   →  공개 저장소·OSS·사내 자산에서 부품 발굴
입고 검사     →  발굴물 5대 KPI 검증
안전성 진단   →  주작-sal-da 방식 위험도 감사 — 결함 부분 식별
해체·선별     →  부품 단위 분해 / 결함 부분 걸러내기 / 검증된 부품만 선별
부품 창고(BOM)→  검증 통과 부품 목록 (5분류: 🟢🟡🔴🏭🔵)
조립          →  검증 부품 + 자체 작성 결합 → SKILL.md
출하 검사     →  4단 QC (정적·동적·호출·외부 검증) + 별도 Verification Agent
출하          →  ~/.claude/skills/ 등재 + CLAUDE.md 기록
운영·개선     →  사용 통계·피드백 → Phase 1 회귀
```

핵심은 **"부품 선별·조립·운영 루프"** — 공급원에서 부품을 받되 결함은 걸러내고 검증된 부품만 가져다 조립하며, 출하 후에도 피드백을 다음 사이클로 환류시킨다.

## 핵심 원칙 (8대 철칙)

1. **발굴물 무신뢰** — 공개 저장소·사내 자산도 통째로 가져다 쓰지 않는다. 반드시 분해·검증한다.
2. **부품 단위 선별** — 발굴물 전체가 아니라 검증된 부품(Phase 블록·프롬프트·체크리스트)만 추출한다.
3. **운용 충돌 0건 — 타협 불가** — PO 글로벌 헌법(한국어 출력·보안 가드·SAL/MBO 자동 감지)을 깨는 부품은 기능이 아무리 좋아도 적용 거부한다.
4. **10라운드 + α 문답 항상 강제** — 막연한 요청을 그대로 받지 않는다. 모드에 맞춰 묻고 답해 스펙을 외부로 끌어낸다.
5. **MBO 승인 게이트** — Phase 2 PO 승인 없이 제조(Phase 3 이후) 시작 금지.
6. **자기검증 금지** — 제조한 스킬을 제조자가 검증하지 않는다. 별도 Verification Agent를 투입한다.
7. **"파일 생성 ≠ 동작함"** — `~/.claude/skills/`에 SKILL.md가 있는 것만으로 완료가 아니다. **실제 슬래시 호출로 의도대로 작동**해야 완료다.
8. **자산화·운영까지가 완료** — 만들고 끝이 아니다. 등재 + CLAUDE.md 기록 + 운영 루프 가동까지 해야 출하 완료다.

### 스킬 특화 4대 원칙 (V3.2 유지)

- **단일 책임** — 하나의 명확한 목적만. 여러 목적을 한 스킬에 욱여넣지 않는다.
- **자기완결** — 만든 스킬은 다른 스킬 참조 링크에 의존하지 않는다. 필요한 내용은 추려서 자기 폴더 안에 참조 자료로 둔다.
- **간결** — 500줄 이내 권장. 거대 스킬 금지.
- **구조 완비** — YAML frontmatter(`description`·`user-invocable`) + Phase 구조 + 사용 예시 + 금지사항 — 4요소.

## Phase 0. 시동 (Intake)

- 인자가 있으면 그것을 제조 대상으로 한다.
- 인자가 없으면 대화 맥락에서 "무슨 스킬을 원하는지 + 단일/배치 중 어느 모드인지" 도출한다.
- 도출 불가 시 PO에게 한 줄 질문으로 확인한다.

### 모드 판별 (V3.2 신설)

| 신호 | 모드 |
|------|------|
| "~하는 스킬 만들어줘" / 1건 / 단일 목적 | **단일 모드** (Phase 1a) |
| "여러 개 묶음" / "한 카테고리 전부" / "공통 패턴" / 표준 양식 N종 | **배치 모드** (Phase 1b) |

> **포털 모드(1c)는 스킬에 무관** — 포털형은 호스팅·인증·DB 인프라가 본질이라 스킬(메타 절차서) 단위에는 적용되지 않는다. 포털형 자산이 필요하면 `/llm-dependent-agent-create`로 가야 한다.

**판별 불가 시 결정 트리 (Q1)**:
```
Q. 만들 스킬 수가 2개 이상이면서 카테고리·패턴이 공통인가?
    NO  → 단일 (Phase 1a)
    YES → 배치 (Phase 1b)
```

**PO 강제 거부권**: PO가 결정 트리 결과에 동의하지 않으면 출하 보고서에 `mode_override: true` + `tree_recommended` + `po_decision` + `risk` 명시.

### 첫 질문 (도출 불가 시)

> *"어떤 스킬을 만들고 싶으십니까? 막연하게 한 줄로 주셔도 됩니다 — 이후 10라운드 + α 문답으로 함께 충분히 기획하겠습니다.
> 그리고 다음 두 가지도 알려주세요:
> ① 제조 모드 — 단일(1개) / 배치(N개 묶음)
> ② **작업 폴더** — 스킬을 어느 폴더에 만들지 (보통 `~/.claude/skills/{name}/`)"*

### 작업 폴더 지정 ★V3.4 필수 절차

**Phase 1 진입 전 PO에게 작업 폴더 위치를 반드시 물어 확정한다.** 표준 위치는 `~/.claude/skills/{name}/`이나 PO 명시 지정도 허용.

## Phase 1. 요구 발굴 — 모드별 10라운드 + α 문답 (V3.5 — 10라운드 기본)

**막연한 요청을 그대로 제조에 넣지 않는다.** 모드에 맞춰 한 번에 한 질문씩 진행한다.
(출처: SUNNY AI MAGIC 50 #01 10라운드 + α 문답 브레인스토밍 → V3.5에서 길이 절반으로 단축)

### ★ V3.5 핵심 원칙 — 질문은 AI가 한다 + 초보자 친화

1. **묻는 주체는 AI(스킬)이다.** PO에게 항목 나열을 던져 일방적으로 답하라고 시키지 않는다. AI가 한 질문씩 던지고 → PO가 답하고 → 다음 질문.
2. **초보자도 답할 수 있는 질문이어야 한다.** 기술 용어를 그대로 쓰지 말고 **풀어쓴 뒤 예시 1~2개**를 동반한다.
3. **객관식·예시형 우선.** 가능하면 `AskUserQuestion` 도구로 2~4개 선택지 + "Other" 직접 입력 형태. 자유 서술은 정말 자유 답이 필요할 때만.
4. **10라운드 기본, 부족 시 추가.** 종료 신호가 안 나오면 라운드를 늘려 보강한다.

질문 톤 예시:

| 항목 | ❌ 나쁜 예 | ✅ 좋은 예 |
|---|---|---|
| 트리거 | "트리거 방식은?" | "이 스킬을 언제 부르고 싶으세요? ① /명령어로 직접 부르기 ② 특정 작업 중 자동 / 예: '/diagram-기본', '코드 작성 중 자동'" |
| 자기완결 | "자기완결 점검?" | "이 스킬이 다른 스킬을 호출해야 하나요, 혼자 끝낼 수 있나요? (V3.4부터 외부 호출 금지가 기본)" |

### Phase 1a — 단일 모드 (10라운드 기본)

> 첫 멘트: *"지금부터 스킬 만들기 진행합니다. 한 번에 한 질문씩 드릴 거예요. 어려운 용어는 풀어서 묻고 예시도 함께 드릴 테니 편하게 답해 주세요."*

**10라운드 3구간** (10은 기준선 — 종료 신호가 안 나오면 추가 라운드로 보강)

| 구간 | 라운드 | 발굴 대상 |
|------|:---:|----------|
| **① 좌표·정체성** | 1~4 | 🎯**목표**(이 스킬이 한 줄로 해결하는 것) · 🏷️**이름**(`/{slash-name}` 형태) · 죽일 수작업·트리거·입력·출력·단일 책임 정의 |
| **② 제약·맥락** | 5~8 | 제약·실패 모드·다단계/단발·🔧호출 도구·🔁절차 루프(1회·반복)·자기완결 점검·★출력 결과가 보일 곳★ |
| **③ 분류·마무리** | 9~10 | 📂**스킬 유형 분류**(절차형/체크리스트형/INSTRUCTION형/도구 조합형/하이브리드) · 차별점·빠진 곳·다음 단계 동의 |

### Phase 1b — 배치 모드 (5라운드 공통 + 변주)

N개 스킬을 1개씩 돌리는 건 비현실적이므로 **공통 + 변주** 구조로 진행:

| 단계 | 라운드 | 발굴 대상 |
|------|:---:|----------|
| **공통 10라운드 + α 문답** | 1~5 | 카테고리 공통 — 입출력 패턴·공통 트리거·공통 도구·공통 제약·★출력 대상 시스템 공통★ |
| **개별 변주** | 6~N | 스킬별 차이점만 짧게 (각 스킬당 1~3라운드) |
| **카탈로그 수립** | 종합 | N개를 표 1장으로 정리 |

### 종료 신호 / 금지

- **종료 신호**: 모드별 산출물(아래)이 모두 채워진 시점.
- **부족 신호 → 라운드 추가**: 산출물 항목 중 비어 있는 게 있거나 PO 답변이 "잘 모르겠다"·"적당히"로 끝났으면 **그 항목만 짚어 1~3 라운드 추가 질문**. 10라운드 채웠다는 이유로 강제 종료 금지.
- **금지**: 라운드 중간에 "바로 답 줘"로 건너뛰지 않는다 — 막연한 스펙으로 회귀한다.

## Phase 2.5 ★V3.5 R15 신설 — 도메인 심층 리서치 (Deep Domain Research)

> **PO 지적 회고 (2026-05-29)**: "AI가 어느 정도 상황 파악한 다음에 지구상에 있는 모든 관련된 것들을 조사하는 그 프로세스가 있어야만이 제대로 된 수준 높은 스킬이 나온다." → Phase 3은 "부품 발굴"이라 스킬 도메인 자체 광역 조사가 빠져 있었음. 형제 공장 R15와 동일 동기화.

**왜 필요한가**: Phase 1·2까지는 "PO 의도 파악". Phase 3은 "부품 단위 발굴". 그 사이에 **"이 분야 자체에 어떤 깊이가 표준인가"** 광역 리서치가 빠지면 표면적 스킬로 끝남.

### 2.5-1. 키워드 망 작성
Phase 1·2에서 도출된 도메인·문제·트리거 키워드를 영문·한국어 양쪽 정리 (핵심 3~5 + 보조 5~10 + 부정 2~3).

### 2.5-2. 4개 트랙 병렬 조사 (★핵심)

| 트랙 | 조사 대상 | 출처 |


**판정 절차**:
1. **출처 우선** — 발굴물에서 가져온 것이면 🟢/🟡
2. **범위 우선** — 자체 작성이지만 N 스킬이 공유한다면 🔵 (배치 모드만)
3. **신규 작성 단독 사용** — 🏭
4. **결함** — 🔴 (다른 분류와 동시 적용 불가)

BOM 작성자는 한 부품을 두 분류에 동시 기입하지 않는다 — 단일 레이블만.

## Phase 5. 설계 검증 — 자기완결 절차 (V3.4)

### ★ 자기완결 원칙 정의 (V3.5 R9 — 두 공장 동기화)

**자기완결(Self-contained)이란**: 본 스킬이 작동하는 동안 다음 3가지를 모두 만족하는 상태.
1. **외부 스킬 호출 금지** — `/pro-persona-debate`·`/주작-sal-da`·`/review-evaluate` 등 다른 user-invocable 스킬을 의무로 호출하지 않는다. 핵심 절차는 본문에 직접 내장.
2. **외부 AI(가족 외 LLM) 호출 금지** — codex CLI(GPT)·Gemini API·Grok 등 Claude 외 LLM을 의무 흐름에 두지 않는다. (단 Phase 7 외부 LLM 교차 검증은 *선택적 외부 게이트*로 제한, 미실시 폴백 명시)
3. **내장 Subagent로 검증** — 검증·평가는 `Agent(subagent_type=..., model="sonnet")` Cross Validator Subagent를 본 스킬이 직접 띄워 수행. 외부 스킬에 위임하지 않는다.

이 정의는 형제 공장 `/llm-dependent-agent-create`와 한 글자 변경 없이 동일 적용된다.

⚠️ **V3.4 변경**: V3.3까지 `/pro-persona-debate` + `/주작-sal-da` 외부 호출 의무였으나, **자기완결 원칙** 적용으로 두 스킬의 핵심 절차를 본문에 내장. 외부 스킬·외부 AI(codex CLI 등) 호출 금지.

**Phase 5 = 5A 검토·평가 + 5B 진단·해체**. S 규모는 5A만, M·L은 둘 다.

### 🔬 Phase 5B. 진단·해체 (SAL Grid 핵심 절차 내장) ★V3.4 신설

**review-evaluate(점수 매기기)와 다른 가치 — SKILL.md 부품을 단위로 해체하고 의존성·결함 지점을 카드별 진단.** M 규모 이상 적용.

#### 5B-1. 역방향 분해 — Grid 완성 (5단계)

```
[1] 구성요소 목록 파악     — BOM 부품 + SKILL.md Phase 블록 단위
[2] 진단 Area 확정          — 4~9개 선택 (구조·자기완결·간결·헌법·실행·문서·재사용 등)
[3] 구성요소 × Area 매트릭스 — 관련성 ≥ 0.6 셀만 유효 진단 단위
[4] 의존성 분석             — Stage 번호 (1, 2, 3...) + Stage 명칭
[5] SAL ID 확정             — S{stage}{AREA}{level} (예: S1SC1 = Stage1·자기완결·Level1)
```

#### 5B-2. Stage별 카드 처리 (의존성 순)

각 카드 = **진단 → 개선안 → 개선 → 재진단** (한 카드에서 완결)

| 단계 | 내용 |
|---|---|
| **진단** | Status (Pass/Warning/Danger/Fail) + Issues + Evidence + Impact |
| **개선안** | Plan + Priority + Expected Effect |
| **개선** | SKILL.md 본문 수정 + Content |
| **재진단** | Re-status + Final Verdict + Remarks |

#### 5B-3. Stage Gate (다음 Stage 진입 조건)
- [ ] 현재 Stage 모든 카드 Final Verdict 확정
- [ ] Fail 카드 0
- [ ] 다음 Stage 의존성 충족

#### 5B-4. 자가 수렴 재진단 루프
Re-status가 Danger·Warning(미허용)이면 같은 카드에서 **재개선 → 재진단** 반복. Pass 수렴까지.

## Phase 7. 출하 검사 — QC 4단 (V3.2)

**자기검증 금지.** 제조에 참여하지 않은 **별도 Verification Agent**를 투입한다.
(출처: SUNNY AI MAGIC 50 #28 / review-evaluate-코어1)

### 7-0. Verification Agent 소환 메커니즘 (V3.2 신설)

"별도 Verification Agent를 투입한다"는 선언만으로는 V1.0처럼 자기검증으로 회귀할 수 있다. 다음 메커니즘을 강제한다:

**소환 절차 (4단계 — 모두 필수)**:

1. **별도 Agent 도구 호출** — `Agent(subagent_type="...", model="sonnet", mode="auto")`로 신규 Subagent를 띄운다.
2. **제조 컨텍스트 비전달** — Verification Subagent prompt에는 SKILL.md와 **검증 기준**만 전달. 제조 과정의 의도·근거·내부 결정 사항은 전달 금지.
3. **외부 검증 도구 사용 의무** — YAML frontmatter 검증·markdown lint·slash 호출 시뮬 등 **외부 도구 결과**를 함께 보고.
4. **호출 증거 기록** — Phase 8 출하 보고서의 "QC 결과" 항목에 `verification_agent_invocations: N` 카운트 + 호출 시각·duration 명시.

**금지**:
- ❌ 제조 Subagent에게 "이제 검증해라"라고 추가 지시
- ❌ 메인 Claude Code(SA)가 직접 검증 의견 작성
- ❌ Subagent 호출은 하지만 prompt에 제조 과정을 다 보여줘서 편향 유도

**위반 시 효과**: QC 결과 무효 처리, Phase 7 처음부터 재실행.

> **V3.4 자기완결**: 외부 스킬(/review-evaluate·/pro-persona-debate·/주작-sal-da) 위임 금지. Cross Validator Subagent를 본 스킬이 직접 띄운다 (Phase 5-3 참조).

- INSTRUCTION 텍스트를 컨텍스트에 주입만 하고 별도 파일·외부 호출 없는 스킬

### 9-2. 피드백 수집

- **이슈 트래커** — GitHub Issues 또는 README의 연락처
- **자동 수집** — 실패 패턴 로그
- **PO 인터뷰** — 분기·반기 단위 정성 피드백

### 9-3. 개선 사이클 → Phase 1 회귀

피드백이 일정 수준 누적되면 다음 사이클을 트리거. 다음 **정량 임계값** 중 하나라도 충족 시:

**자동 트리거 조건 (OR)**
- **사용률 0** — 출하 후 30일간 호출 0회 (폐기/통합 검토)
- **반복 실패** — 같은 오류 ≥ 3회 (직전 7일)
- **피드백 누적** — "버그·개선요청" ≥ 3건 (직전 14일)
- **PO 정기 점검** — 매월 1일 운영 KPI 리뷰 (강제)

**트리거 발동 후 회귀 깊이 매핑**:

| 트리거 신호 | 회귀 깊이 |
|--------|----------|
| 단순 문구·예시 수정 (피드백) | Phase 6 재조립만 |
| 부품 변경·새 절차 추가 (반복 실패) | Phase 4 재분해부터 |
| 정책·구조 변경 (정기 점검) | Phase 2 MBO 재승인부터 |
| 사용률 0 스킬 폐기 결정 | Phase 8 자산 갱신 + Phase 9 폐기 절차 |

### 9-4. 점진적 배포 (Canary)

L 규모 스킬은 다음 사이클 산출물을 한꺼번에 풀지 않는다:
- **Canary** — PO 본인이 1주 선행 사용 → 안정 확인
- **Rollout** — 팀 또는 공개 배포
- **Rollback** — 문제 발견 시 즉시 되돌릴 수 있는 경로 (이전 버전 폴더 백업)

### 운영 루프 종료 조건

스킬 폐기·교체·통합 시점까지 무한 루프. 폐기 시:
- CLAUDE.md 등재 갱신 ("폐기" 또는 "후속 스킬로 이관")
- 사용 로그 아카이브

## /llm-dependent-agent-create와의 관계 (V3.2 — 양방향 명시, V3.5 후속 — 경계 표 강화)

`/llm-dependent-agent-create`는 형제 공장이다. 같은 9-Phase 방식·같은 분해·검증 후 조립 철학을 공유한다.

### ★ 두 공장 경계 — 어느 쪽으로 가야 하나? (V3.5 후속 신설)

| 차원 | `/skill-create` | `/llm-dependent-agent-create` |
|---|---|---|
| **제조 산출물** | `~/.claude/skills/{name}/SKILL.md` (마크다운) | 운영 가능한 에이전트 자산 (코드·UI·DB·인프라) |
| **실행 주체** | Claude Code가 컨텍스트에 SKILL.md를 주입해 실행 | LLM API + 엔진 코드 + 사용자 인터페이스 |
| **사용 위치** | 개발자 본인의 Claude Code 세션 | 외부 사용자가 호출하는 서비스 (웹·CLI·메신저 등) |
| **DB 필요** | 없음 (마크다운만) | A 인프라(Supabase) 또는 B 인프라(JSON 로컬) |
| **호스팅** | 로컬 `~/.claude/skills/` 폴더 | Vercel·Cloud Run·로컬 PC·메신저 봇 등 |
| **모드** | 단일 / 배치 (포털 불가 — 인프라 없음) | 단일 / 배치 / **포털** (1 호스팅 = N 에이전트) |
| **인프라 분기** | 없음 (마크다운에 인프라 없음) | A (Supabase) / B (DB 없음, .py + .env) |
| **Phase 6 구조** | 단일 — SKILL.md 1장 조립 | 3단 — 6a 본체 + 6b 플랫폼(포털만) + 6c 공통 인프라(배치·포털만) |
| **5요소 적용** | 자기완결·단일 책임 중심 | 자기완결·자율 루프·메모리·도구·자가평가·헌법 5요소 전부 |
| **Phase 7-3+ 사람 눈 검증** | 외부 시스템 출력 스킬만 (대다수 면제) | 거의 모든 에이전트 (외부 출력 본질) |
| **Phase 8 Windows 어댑터 (R7·R8)** | standalone `.py`+`.bat` 배포 스킬만 적용 (대다수의 SKILL.md 전용 스킬은 면제) | 인프라 B + Windows + 사람 수동 트리거 경로 있는 에이전트 의무 — install_shortcut.ps1 자동 제조 + 사전 검증 + 선택적 SmokeTest |

### 어느 쪽으로 갈지 결정 (5초 가이드)

```
산출물이 "Claude Code 컨텍스트 안에서 절차·체크리스트·INSTRUCTION으로 동작"하는가?
├─ YES → /skill-create (이 공장)
│         ├─ 단일 스킬 1개 → Phase 1a 단일 모드
│         └─ N개 스킬 묶음 → Phase 1b 배치 모드
│         예: /code-review-기본, /diagram-기본, /review-evaluate-코어1
│
└─ NO → 외부 사용자가 직접 호출하는 서비스가 필요한가? → /llm-dependent-agent-create (형제 공장)
        ├─ 외부 호출자 1유형·1 호스팅 → 단일 모드 (1a)
        ├─ N개 묶음·각자 호스팅 → 배치 모드 (1b)
        └─ 1 호스팅 = N 에이전트 + 공통 인증·관리 → 포털 모드 (1c, 인프라 A 필수) ★V3.5 R9 명시
        예: 대규모 사내 에이전트 포털(N≥50, 포털 모드), 뉴스 요약 에이전트(단일 B), 카톡봇·Slack봇(단일 A)
```

**양방향 변환 가능** — 같은 도메인 지식을 두 형태로 포장 가능:
- 스킬 → 에이전트: SKILL.md의 INSTRUCTION을 에이전트의 system_prompt로 재포장
- 에이전트 → 스킬: 에이전트 system_prompt를 SKILL.md로 추출 (개인 사용으로 축소)

대규모 사례: N개 도메인 스킬(`스킬스/*/SKILL.md`)이 동시에 N개 LLM 에이전트의 system_prompt로도 사용됨. 같은 자산 두 포장. (사내 에이전트 포털 운용 사례에서 확인)

**역할 분담 한 줄**:
- `/skill-create` = **도메인 지식의 마크다운화** (한 절차·체크리스트·INSTRUCTION을 SKILL.md로)
- `/llm-dependent-agent-create` = **그 마크다운을 LLM이 실행하는 에이전트로 패키징** (엔진 + UI/CLI + 인프라)

## 금지 사항 (안티 패턴 V3.2 — 10개)

❌ **절대 하지 말 것**:

1. **통째 채택** — 발굴물을 분해 없이 그대로 가져다 쓰지 않는다 (Phase 4 필수).
2. **결함 적용** — 🔴 판정 부품을 "기능이 좋으니까"로 되살리지 않는다.
3. **운용 충돌 묵인** — 헌법 위반 부품을 글로벌에 들이지 않는다 (타협 불가).
4. **10라운드 + α 문답 생략** — "요청이 명확하다"는 자가 판단으로 Phase 1을 건너뛰지 않는다.
5. **MBO 게이트 무시** — PO 승인 없이 제조 시작 금지.
6. **자기검증** — 제조자가 자기 스킬을 QC하지 않는다.
7. **"파일 생성" 합격 처리** — ~/.claude/skills/에 파일이 있다는 것만으로 출하하지 않는다. 실제 호출로 검증.
8. **거대 스킬 + 외부 참조 의존** — 단일 책임 위반·자기완결 위반 금지. 500줄 권장 초과 시 분할 검토.
9. **모드 무선언** (V3.2) — 단일/배치 모드를 명시하지 않고 진행하지 않는다.
10. **운영 루프 단절** (V3.2) — Phase 9 진입 없이 "출하 완료"로 종결하지 않는다.
